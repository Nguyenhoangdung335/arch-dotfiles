name: code-review
description: Perform a structured, multi-phase code review for Rust projects focusing on quality, safety, and testing.
---

Your primary objective is to systematically analyze, evaluate, and review code modifications in the application. This workflow leverages best practices for structured code reviews.

## Phase 1: Review Preparation

### 1. Gather Context

- **Change Intent**: Understand the problem or feature the code attempts to address.
- **Codebase Structure**: Review how the changes fit into the existing architecture and dependencies.
- **Related Issues**: Identify any linked bug reports or feature requests.
- **Test Coverage**: Note whether new tests were added or existing ones modified.

### 2. Automated Validation

- **Format Check**: Run `cargo fmt --check` to ensure adherence to Rust formatting standards.
- **Linting**: Execute `cargo clippy --all-targets --all-features` to catch common mistakes and unidiomatic patterns.
- **Test Execution**: Run `cargo test` to confirm no existing logic is broken by the changes.
- **Tooling Outputs**: Collect and review any warnings or errors generated by the tooling.

## Phase 2: Deep Inspection

### 3. Code Quality Analysis

- **Ownership & Borrowing**: Verify correct use of references; flag unnecessary `.clone()` or allocations.
- **Idiomatic Rust**: Check for proper use of iterators, traits, and generics over manual loops or over-engineering.
- **Data Flows**: Understand how data moves through the new code to ensure efficiency.
- **Anti-Patterns**: Check for common issues like over-abstraction, hidden macro logic, or global mutable state.

### 4. Safety & Error Analysis

- **Error Propagation**: Ensure errors are handled gracefully (e.g., using `?` and `Result`) rather than swallowed.
- **Panic Prevention**: Strictly flag `.unwrap()`, `.expect()`, `unreachable!()`, or `panic!()` in production code.
- **Unsafe Code**: Scrutinize `unsafe` blocks for soundness; require `# Safety` documentation and suggest safe alternatives.
- **Concurrency**: Identify potential race conditions, deadlocks, or improper shared state (e.g., `Arc<Mutex<T>>` or `Arc<RwLock<T>>`).

## Phase 3: Architecture & Testing

### 5. Design Assessment

- **Modularity**: Assess if large files, functions, or structs should be split into smaller, focused components.
- **Visibility**: Ensure correct use of `pub` to encapsulate internal logic and minimize the public API surface.
- **Defensive Programming**: Verify that the code includes checks to prevent invalid states.
- **Documentation**: Code must have doc comments (`///`) for public functions, structs, and modules.

### 6. Test Verification

- **Test Validity**: Review added tests to ensure they accurately test the new logic.
- **Edge Cases**: Check if tests cover unusual, extreme, or boundary inputs.
- **Regression Checks**: Ensure the changes do not introduce regressions into existing functionality.
- **Mocking & Fixtures**: Verify that test setups are clean and do not rely on flaky external state.

## Phase 4: Feedback & Reporting

### 7. Formulate Feedback

- **Actionable Fixes**: Provide concrete code examples for every issue or improvement pointed out.
- **Constructive Tone**: Ensure feedback is objective, polite, and focuses on the code, not the author.
- **Prioritization**: Clearly differentiate between mandatory blocking issues and optional nitpicks.
- **Consistency**: Align recommendations with existing codebase patterns and idiomatic Rust guidelines.

### 8. Final Review Report

- **Summary**: Concise evaluation of the overall changes and their impact.
- **Critical Issues (Blockers)**: Document bugs, memory leaks, panics, concurrency flaws, or security vulnerabilities.
- **Warnings & Smells**: List unidiomatic code, unnecessary allocations, or unresolved clippy warnings.
- **Suggestions (Nitpicks)**: Propose minor refactors, better naming conventions, or slight performance tweaks.
- **Positive Feedback**: Highlight good practices, clever solutions, and well-written tests.
